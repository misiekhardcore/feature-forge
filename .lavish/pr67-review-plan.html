// In execute():
const reporter = ctx.ui ? new TuiProgressReporter({ ctx, onStateChange: ... })
                         : new NoOpProgressReporter();

eventBus.on("feature-forge:*", (data) => {
  const event = RoutineTool.toProgressEvent(data, ...);
  
  // Accumulate agent state (moved from TuiProgressReporter)
  this.agentState.set(event.agentId, { status: event.agentStatus, summary: event.agentSummary });
  
  // Build pre-fabricated content — all theme colors applied here
  const theme = ctx.ui.theme;
  const title = `${theme.fg("accent", "⟳")} ${this.routineName}`;
  const subtitle = event.maxIterations > 0
    ? `iteration ${event.iteration + 1}/${event.maxIterations}`
    : undefined;
  
  const rows = [...this.agentState].map(([id, a]) => ({
    icon: ProgressRenderer.statusIcon(a.status, theme),
    label: id,
    annotation: a.summary,
  }));
  
  const widgetLines = ProgressRenderer.buildWidgetLines({
    theme, title, subtitle, rows,
    metadata: event.continueWhile
      ? [{ key: "while", value: event.continueWhile }] : undefined,
    path: event.workspace,
  });
  
  const statusText = ProgressRenderer.buildStatusLine({
    theme, title, subtitle,
    tags: [...this.agentState].map(([id, a]) => ({
      icon: ProgressRenderer.statusIcon(a.status, theme),
      label: id,
    })),
  });
  
  reporter.render(widgetLines, statusText);
});

Open Questions

Do all routines have iterators? Should formatIterationBadge live in ProgressRenderer or be caller-built?

Decision: No — only looped routines have iterators. RoutineTool.findLoopConfig() already detects this: when maxIterations === 0, the routine has no loop, and the caller passes "" (empty string) as iterationBadge. The widget/status renderers then omit the iteration segment. ProgressRenderer never sees loop concepts — it just renders whatever badge string it receives.

Should we use ToolRenderer instead of creating ProgressRenderer?

Decision: No — separate the two. ToolRenderer renders tool call/result rows in the chat (spawnAgentCall, sendTaskCall, etc.). ProgressRenderer renders widget panels and status bars — a different rendering surface with different content shape. Renaming ToolRenderer to encompass both would violate single responsibility. They share the same pattern (exported pure functions, theme-aware), which is the right design — just in different modules.

Should onUpdate in RoutineTool switch to Partial<RoutineResult>?

Decision: Yes. The TDetails becomes RoutineResult, and the onUpdate callback emits Partial<RoutineResult> since progress events carry partial data. The update callback's details use a cast to RoutineResult with only populated fields.

Do we need a shared ProgressState accumulator or can RoutineTool own it all?

Proposal: RoutineTool owns agent state accumulation as a private Map<string, { status, summary }>. If another tool needs the same accumulation logic, we extract it later. YAGNI for now.
